Fix:
  Root cause: The original approach called portraitFileInput.click() programmatically on a display:none element from
  within a JS event listener. While this technically works in Chrome/Edge, it's fragile — browsers treat programmatic
  .click() on hidden inputs inconsistently, and it can be silently blocked depending on browser version, security
  context, or timing.

  Fix: Moved the <input type="file"> inside the .portrait-area div and gave it a new .portrait-file-input CSS class that
   positions it as a transparent (opacity: 0) full-size overlay at z-index: 2. The user now clicks directly on the input
   itself — the browser opens the file picker natively with no JavaScript dispatch involved at all. The change event
  handler in JS is unchanged; only the click wiring was removed. The img stays at z-index: 1 so the input overlay stays
  on top even after a portrait is loaded, allowing the user to click again to change it.
